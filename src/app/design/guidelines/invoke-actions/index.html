Two interacting principles guide SKY UX hyperlinks and buttons: style and implementation. Style determines the appropriate appearance of an affordance to the user, while implementation involves the core HTML decision to use the <a> element or the <button> element. In SKY UX, these guidelines overlap and interact: Differences in styling do not need to correspond to the use of different tags.

Style

SKY UX uses several styles for hyperlinks and buttons. The differences in appearance carry meaning about roles in the system, distinguish between navigation and action, or convey different levels of emphasis or priority.

Primary actions

The style for a primary action is a bold, saturated color that conveys importance and drives user workflows. Only use one primary action in any given system state. Use primary actions for the most important action on a page, the most frequently used action on a page, or the progressive terminal action on a dialogue (such as 'Save' or 'Done'). For information about how to use primary actions, see the primary action guidelines.

Secondary actions

Secondary actions are less important or less frequently used than primary actions, so SKY UX reduces the visual weight to make them noticeable but not attention-drawing. Use secondary actions for:

  • Toolbar or action bar buttons

  • Actions that are relevant to only part of a page’s content, such as actions in cards or tiles

Tertiary actions

Tertiary actions receive a lighter visual weight but invisibly occupy the same amount of space as primary and secondary actions. Use tertiary actions for:

  • Infrequently used actions or options on a page

  • The “regressive” action on a dialogue, such as a cancel action

Inline links

Inline links receive the least visual weight and appear inline with text instead of in their own rectangular space. They have no button padding or spacing and should not be used in groups with other buttons. Do not “stack” multiple inline links too close together because this creates a usability risk of mis-clicks or mis-touches. Use inline links for pure navigation:

  • Parent-to-parent

  • Parent-to-child

  • Links to external applications or services, such as a street address that links to a map application

Implementation

While SKY UX styling properties allow <a> and <button> elements to appear interchangeable, subtle differences in their behavior make it important to select the correct implementation. Most notably, these elements differ in:

  • Right-click options

  • Long-touch and force-touch behaviors for mobile browsers

  • Keyboard shortcuts

  • Behavior with screen readers and other accessibility software

To determine the correct HTML tag for an affordance in SKY UX, follow this guidance:

  1. If the ONLY result when users click, tap, or select the affordance is loading a new URL in the browser, then implement the affordance as an <a> element.

  2. Otherwise, implement the affordance as a <button> element.

In other words, <a> elements are only appropriate when an affordance changes nothing about the state of the system and the only result is that the user traverses from one view of content to another. For example, a record page may contain an affordance to delete the record. If a user activates this affordance, the browser redirects to a new URL because the existing page presumably ceases to exist. However, since this also impacts the system by deleting data, you should implement the affordance as a <button> element.

Primary action guidelines

No matter the page layout, SKY UX designs should follow consistent guidelines for the placement and styling of primary actions.

  • Primary actions belong in the top-left of pages underneath the page titles.
    • For some page types, primary actions are placed within containers on the page. For details, see "How to use primary actions" below.
  • Pages should only include 1 primary action so that it stands out.
    • The primary action should be the most important or most frequently used action on a page.
      • Common examples include adding a new child record, or performing an important status change on a record page.
    • If pages have child records with important actions, don't expose all of them as primary actions. Even if actions are primary to child records, they cannot all be primary to the parent record.
    • Many pages have no primary action.
      • Don't force it - if it isn't fairly obvious what a page's primary action should be, it probably doesn't have one.
    • If users can accomplish a primary action in multiple ways, don't expose all of them as primary actions.
      • Instead, present that decision after the user has started taking the primary action (see Action button).
    • Use SKY iconography when appropriate.
How to use primary actions

For work centers or other pages that do not have page summaries, primary actions belong in the top-left under page titles and above all other content.

For work centers or other pages with page summaries, primary actions belong under page titles inside the summary. Other content in the summary belongs to the right or under the primary action.

For work centers or other pages with page summaries and page-level toolbars, primary actions belong in the leftmost position inside the page toolbar.

For list pages that contain a single list, primary actions belong in the leftmost position inside the list toolbar. Likewise, if pages consist entirely of tabs and have no other content, primary actions on tabs that contain a single list belong in the leftmost position inside the list toolbar.

How not to use primary actions

Do not use multiple primary actions on pages.

Do not use primary actions for child-specific actions on parent pages. The only exception is for adding child records.

Do not use other button styles or arrangements to represent primary actions.

Do not use primary buttons outside of page-level toolbars when pages include them.
